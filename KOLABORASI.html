Apa itu GitHub & Kenapa Dipakai?

Penjelasan sederhana sebelum mulai

Analogi Sederhana

Bayangkan project PanelFlow seperti dokumen Google Docs yang dikerjakan bareng. Masalahnya, kode program tidak bisa diedit bareng secara real-time seperti Google Docs — kalau dua orang ubah file yang sama bersamaan, hasilnya akan kacau.


GitHub adalah solusinya. Cara kerjanya: setiap orang mengambil salinan project ke laptop masing-masing, mengerjakan bagian mereka, lalu mengajukan perubahan untuk diperiksa sebelum masuk ke project utama.

Keuntungan
  • Tidak ada pekerjaan yang hilang
  • Bisa lihat siapa ubah apa & kapan
  • Bisa kembali ke versi sebelumnya
  • Tim bisa kerja paralel tanpa bentrok
Istilah Penting
  • Repository (Repo) = Gudang project
  • Branch = Jalur kerja terpisah
  • Commit = Simpan perubahan
  • Pull Request = Ajukan perubahan
  • Merge = Gabungkan ke utama

Setup Pertama Kali

Lakukan ini sekali saja saat pertama kali bergabung ke tim

1
Install Git di Laptop
Git adalah alat yang dipakai untuk berkomunikasi dengan GitHub dari terminal / command prompt.
  • Mac: Buka Terminal → ketik git --version. Kalau belum ada, ikuti instruksi yang muncul.
  • Windows: Download dari git-scm.com → install dengan klik Next terus.
  • Linux: Ketik sudo apt install git di terminal.
2
Perkenalkan Diri ke Git
Git perlu tahu namamu dan emailmu untuk mencatat siapa yang mengubah apa.
Terminal 
# Ganti dengan nama dan email GitHub kamu
$ git config --global user.name "Nama Kamu"
$ git config --global user.email "email@kamu.com"

# Verifikasi sudah benar
$ git config --list
3
Clone Repository PanelFlow
Ini artinya "download salinan project ke laptop kamu". Lakukan di folder yang kamu inginkan.
Terminal 
# Masuk ke folder tempat kamu mau taruh project
$ cd ~/Desktop

# Clone project (salin URL dari GitHub)
$ git clone https://github.com/nama-org/PanelFlow.git

# Masuk ke folder project
$ cd PanelFlow

# Install semua dependencies
$ npm install
4
Setup File .env
File ini berisi API key rahasia. Tidak pernah di-upload ke GitHub — harus dibuat manual.
Terminal 
# Salin template .env yang sudah tersedia
$ cp .env.example .env

# Buka file .env dan isi API key kamu
# GEMINI_API_KEY=isi_api_key_kamu_di_sini
⚠️
Jangan pernah share isi file .env ke siapapun atau commit ke GitHub. File ini sudah terlindungi oleh .gitignore.
5
Test Jalankan Project
Pastikan semuanya berjalan normal di laptop kamu.
Terminal 
$ npm run dev

# Buka browser ke: http://localhost:3000
# Kalau muncul tampilan PanelFlow AI, setup sudah berhasil! 🎉

Membuat Branch Baru

Branch = jalur kerja terpisah agar tidak mengganggu kode utama

Apa itu Branch?

Bayangkan project kamu seperti pohon. Branch main adalah batang utama — inilah kode yang sudah berjalan stabil dan dipakai semua orang.

Saat kamu mau tambahkan fitur baru atau perbaiki bug, kamu membuat cabang baru dari batang itu. Kamu bebas bekerja di cabang sendiri tanpa merusak batang utama. Kalau sudah selesai dan sudah diperiksa, barulah cabangmu digabungkan ke batang utama.

main
← kode stabil
feat/tambah-fitur-x
← kamu kerja di sini
Konvensi Nama Branch
TipeFormat NamaContoh
Fitur Barufeat/nama-fiturfeat/export-mp4
Perbaikan Bugfix/nama-bugfix/panel-detection-crash
Refactorrefactor/nama-bagianrefactor/storage-service
Docsdocs/nama-docdocs/readme-update
✅
Gunakan huruf kecil dan tanda minus (-) sebagai pemisah. Hindari spasi dan karakter khusus.
1
Pastikan kamu di branch main & kode terbaru
Sebelum buat branch, selalu ambil kode terbaru dari GitHub terlebih dahulu.
Terminal 
# Pindah ke branch main
$ git checkout main

# Ambil & gabungkan perubahan terbaru dari GitHub
$ git pull origin main

# Hasilnya harus: Already up to date. atau daftar file yang berubah
2
Buat branch baru & langsung pindah ke sana
Satu command ini langsung membuat branch baru sekaligus pindah ke dalamnya.
Terminal 
# Format: git checkout -b nama-branch-kamu
$ git checkout -b feat/nama-fitur-kamu

# Contoh nyata:
$ git checkout -b feat/export-mp4
# Output: Switched to a new branch 'feat/export-mp4'

# Verifikasi kamu sudah di branch yang benar
$ git branch
# Output: * feat/export-mp4  ← tanda bintang = branch aktif
#           main
3
Mulai bekerja!
Sekarang kamu bebas mengubah, menambah, atau menghapus file apa saja. Kode di branch main tidak akan terpengaruh sama sekali.
💡
Jalankan npm run dev untuk mulai coding sambil melihat hasilnya langsung di browser.

Rutinitas Harian

Yang harus dilakukan setiap kali mulai dan selesai kerja

☀️
Setiap pagi sebelum mulai coding: selalu pull kode terbaru! Ini memastikan kamu tidak ketinggalan perubahan dari teman tim.
A
Ambil kode terbaru (setiap pagi / sebelum mulai)
Terminal — Rutinitas Pagi 
# 1. Pindah ke main dulu
$ git checkout main

# 2. Ambil perubahan terbaru dari GitHub
$ git pull origin main

# 3. Kembali ke branch kerja kamu
$ git checkout feat/nama-fitur-kamu

# 4. Gabungkan update main ke branch kamu (opsional, sangat dianjurkan)
$ git merge main
B
Simpan perubahan dengan commit (berkala saat coding)
Commit = checkpoint. Lakukan setiap kali kamu selesai mengerjakan sesuatu yang berarti. Jangan nunggu sampai semua selesai baru commit!
Terminal — Commit Perubahan 
# 1. Lihat file apa saja yang sudah kamu ubah
$ git status

# 2a. Tambahkan semua file yang berubah ke "antrian commit"
$ git add .

# 2b. ATAU tambahkan file tertentu saja (lebih aman)
$ git add src/services/storage.ts

# 3. Simpan dengan pesan yang jelas
$ git commit -m "feat: tambahkan export video ke format MP4"
Format Pesan Commit yang Baik
AwalanDipakai untukContoh
feat:Fitur barufeat: tambah tombol import project
fix:Perbaikan bugfix: panel hilang setelah auto-snap
refactor:Perbaikan struktur koderefactor: pisah storage ke service sendiri
style:Tampilan / CSSstyle: perbaiki warna tombol navbar
docs:Dokumentasidocs: tambah panduan kolaborasi
chore:Setup / configchore: install jszip dan idb
⬆️

Push ke GitHub

Upload pekerjaan kamu agar bisa dilihat dan di-review tim

Commit vs Push — Apa Bedanya?
💾 Commit

Menyimpan perubahan di laptop kamu. Seperti "Save" di game. Hanya ada di laptop kamu sendiri.

☁️ Push

Mengirim commit ke GitHub. Baru setelah push, tim lain bisa melihat pekerjaanmu.

1
Push branch kamu ke GitHub
Terminal 
# Push pertama kali (branch baru)
$ git push -u origin feat/nama-fitur-kamu

# Push selanjutnya (sudah pernah push sebelumnya)
$ git push

# Output yang berarti sukses:
# Branch 'feat/nama-fitur-kamu' set up to track remote branch
# To https://github.com/.../PanelFlow.git
# * [new branch] feat/nama-fitur-kamu -> feat/nama-fitur-kamu
2
Buka GitHub dan buat Pull Request
Setelah push, GitHub biasanya langsung menampilkan tombol hijau "Compare & pull request". Klik tombol itu!
✅
Kalau tidak muncul, buka tab Pull Requests di halaman repo GitHub dan klik "New pull request".
📋

Membuat Pull Request

Cara mengajukan perubahan kamu untuk diperiksa tim sebelum masuk ke main

Pull Request itu Apa?

Pull Request (PR) adalah cara kamu "mengajukan lamaran" agar perubahan di branch-mu diterima dan digabungkan ke branch main. Di sinilah tim bisa melihat apa yang kamu ubah, memberikan komentar, dan menyetujuinya.

1
Isi judul PR yang jelas
Judul yang baik langsung menjelaskan apa yang dilakukan tanpa perlu buka filenya.
❌ Judul yang buruk:
Update kode
✅ Judul yang baik:
feat: tambahkan IndexedDB untuk storage project + export ZIP .panelflow
2
Isi deskripsi PR
Gunakan template berikut agar reviewer mudah memahami apa yang kamu kerjakan.
Markdown — Template Deskripsi PR 
## 📝 Ringkasan
Jelaskan dalam 2-3 kalimat apa yang kamu ubah dan kenapa.

## 🔧 Perubahan yang Dibuat
- Tambahkan `src/services/storage.ts` untuk IndexedDB
- Ganti localStorage dengan IndexedDB di App.tsx
- Tambahkan tombol Export / Import project di navbar

## 🧪 Cara Test
1. Jalankan `npm run dev`
2. Upload chapter komik
3. Reload halaman → data masih ada
4. Klik tombol Export → file .panelflow terdownload

## ⚠️ Hal yang Perlu Diperhatikan
- Jika ada data lama di localStorage, akan otomatis migrasi
- File .panelflow bisa berukuran besar jika chapter banyak
3
Assign reviewer & label
Di sidebar kanan GitHub, atur:
  • Reviewers: Tag teman yang harus review kodenya
  • Assignees: Namamu sendiri (yang mengerjakan)
  • Labels: Pilih feature, bug, atau enhancement
4
Klik "Create Pull Request"
PR kamu sudah aktif! Reviewer akan mendapat notifikasi email dari GitHub.
💡
Kalau PR masih dalam progress dan belum siap di-review, tambahkan prefix [WIP] di judulnya atau klik "Convert to Draft".
👀

Cara Me-review Kode Orang Lain

Panduan untuk reviewer — cara memberi feedback yang membangun

1
Buka tab "Files changed" di PR
Di sini kamu bisa melihat semua perubahan. Baris merah = dihapus, baris hijau = ditambah.
2
Klik baris kode untuk berkomentar
Hover ke nomor baris → muncul ikon + biru → klik untuk tambah komentar spesifik di baris itu.
Tips Komentar yang Membangun
  • ❌ "Kodenya salah" → ✅ "Bagian ini bisa menyebabkan error jika array kosong, pertimbangkan cek .length dulu"
  • ❌ "Jelek" → ✅ "Bisa lebih readable kalau variabelnya diberi nama yang lebih deskriptif, misalnya panelImages bukan data"
  • Awali dengan hal positif sebelum kritik
  • Bedakan antara saran (boleh diabaikan) vs wajib diperbaiki
3
Submit review dengan status
Klik "Review changes" di pojok kanan atas, lalu pilih:
✅ Approve

Kode sudah baik, setuju untuk di-merge.

🔄 Request Changes

Ada yang perlu diperbaiki dulu sebelum bisa di-merge.

🔁

Merespons Feedback (untuk Penulis PR)

Setelah reviewer memberi komentar, ini yang harus dilakukan

1
Perbaiki kode di laptop
Buka file yang dikomentari, perbaiki sesuai saran reviewer.
2
Commit dan push lagi
Terminal 
$ git add .
$ git commit -m "fix: perbaiki feedback dari review — cek array kosong"
$ git push

# Commit baru ini otomatis masuk ke PR yang sama!
# Tidak perlu buat PR baru.
3
Resolve komentar di GitHub
Buka kembali PR di GitHub, klik "Resolve conversation" di setiap komentar yang sudah kamu tangani. Ini membantu reviewer tahu mana yang sudah beres.
✅

Merge & Approve

Tahap terakhir — menggabungkan perubahan ke branch main

ℹ️
Siapa yang bisa merge? Di project PanelFlow, hanya Owner / Maintainer yang boleh merge ke branch main. Developer tidak boleh merge PR sendiri.
1
Syarat PR bisa di-merge
  • ✅ Minimal 1 reviewer sudah Approve
  • ✅ Semua komentar "Request Changes" sudah di-resolve
  • ✅ Tidak ada konflik dengan branch main
  • ✅ CI/CD checks passed (kalau ada)
2
Pilih metode merge yang tepat
MetodeCocok untukEfek di history
Squash and merge ⭐ Fitur kecil, hotfix Semua commit jadi 1 commit rapi di main
Merge commit Fitur besar, banyak commit penting Semua commit masuk + 1 commit merge
Rebase and merge Advanced users saja Commit diletakkan ulang tanpa merge commit
⭐
Rekomendasi untuk PanelFlow: pakai Squash and merge agar history di main tetap bersih dan mudah dibaca.
3
Klik "Merge pull request" → Confirm
Perubahan sekarang sudah masuk ke branch main! Semua anggota tim bisa pull untuk mendapatkan update terbaru.
4
Hapus branch lama (opsional tapi dianjurkan)
Setelah merge, GitHub akan menawarkan tombol "Delete branch". Klik untuk menjaga repo tetap rapi.
Terminal — Bersih-bersih setelah merge 
# Ambil update main terbaru
$ git checkout main
$ git pull origin main

# Hapus branch lokal yang sudah di-merge
$ git branch -d feat/nama-fitur-kamu

# Hapus juga di GitHub (remote)
$ git push origin --delete feat/nama-fitur-kamu
⚔️

Konflik Kode — Jangan Panik!

Konflik terjadi saat dua orang ubah baris yang sama. Ini normal dan bisa diselesaikan.

⚔️
Konflik bukan berarti salah satu orang salah. Ini hanya berarti Git bingung versi mana yang harus dipakai. Tugasmu adalah memberitahu Git versi mana yang benar.
Tampilan Konflik di File
Contoh konflik di file 
<<<<<<< HEAD (perubahan kamu)
  const voiceId = 'Kore';
======= (pemisah)
  const voiceId = 'Puck';
>>>>>>> feat/tambah-voice-selector (perubahan orang lain)
Kamu harus memilih salah satu (atau gabungkan keduanya) lalu hapus semua tanda <<<, ===, dan >>>.
1
Update branch kamu dengan main terbaru
Terminal 
# Pastikan kamu di branch kerja kamu
$ git checkout feat/nama-fitur-kamu

# Ambil main terbaru
$ git fetch origin

# Coba gabungkan main ke branch kamu
$ git merge origin/main

# Jika ada konflik, Git akan memberitahu file mana yang bermasalah
2
Buka file yang konflik di VS Code
VS Code akan menampilkan tombol pilihan yang memudahkan: "Accept Current Change", "Accept Incoming Change", atau "Accept Both Changes".
💡
Kalau bingung harus pilih yang mana, tanya ke teman yang menulis kode yang konflik tersebut. Jangan asal pilih!
3
Selesaikan konflik dan commit
Terminal 
# Setelah semua konflik diselesaikan di editor
$ git add .

# Commit untuk menyelesaikan merge
$ git commit -m "chore: resolve merge conflict di App.tsx"

$ git push
⏪

Batalkan Kesalahan

Jangan panik. Hampir semua kesalahan di Git bisa diperbaiki.

Situasi 1: Belum commit, mau batalkan perubahan file
Terminal 
# Batalkan perubahan di satu file tertentu
$ git checkout -- src/App.tsx

# Batalkan SEMUA perubahan (hati-hati, tidak bisa di-undo!)
$ git checkout -- .
Situasi 2: Sudah commit, mau batalkan commit terakhir (tapi simpan perubahannya)
Terminal 
# Batalkan commit terakhir, perubahan tetap ada di file
$ git reset --soft HEAD~1
Situasi 3: Pesan commit salah, mau ganti
Terminal 
# Ganti pesan commit terakhir (hanya jika belum di-push!)
$ git commit --amend -m "feat: pesan yang benar"
🚨
Jangan gunakan --amend kalau commit sudah di-push ke GitHub! Ini bisa merusak history orang lain.
Situasi 4: Mau lihat semua commit untuk kembali ke titik tertentu
Terminal 
# Lihat daftar commit dengan ID-nya
$ git log --oneline
# Output contoh:
# a3f2c1b feat: tambah export ZIP
# 7e9d4a0 fix: panel detection error
# 2b8f1c3 chore: install idb dan jszip

# Kembali ke commit tertentu (buat branch baru dari sana)
$ git checkout -b recovery/rollback 7e9d4a0
